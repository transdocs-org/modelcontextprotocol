Model Context Protocol（MCP）遵循正式的治理模型，以确保透明的决策过程和社区参与。本文档概述了项目的组织方式以及决策的制定方式。

## 技术治理

MCP 项目采用层级结构，类似于 Python、PyTorch 和其他开源项目：

- 由 **贡献者** 组成的社区，他们提交问题报告、发起拉取请求并为项目做出贡献。
- 一小部分 **维护者** 负责 MCP 项目中的组件，例如 SDK、文档等。
- 贡献者和维护者由 **核心维护者** 监督，他们负责项目的整体方向。
- 核心维护者中有两位 **首席核心维护者**，是最终的决策者。
- 维护者、核心维护者和首席核心维护者共同组成 **MCP 指导小组**。

所有维护者都应强烈认同 MCP 的设计理念。技术治理流程的成员资格属于个人，而非公司。也就是说，没有为特定公司保留的席位，成员资格与个人有关，而不是与雇佣该个人的公司有关。这确保了维护者以协议本身和开源社区的最佳利益为出发点行事。

### 通信渠道

技术治理通过所有 **维护者、核心维护者** 和 **首席维护者** 共享的 [Discord 服务器](/community/communication#discord) 进行。每个维护者组可以选择额外的沟通渠道，但所有决策及其讨论内容必须记录并在 Discord 服务器上公开。

### 维护者

维护者负责 MCP 项目内的个别项目或技术工作组。这些通常是独立的仓库，如语言特定的 SDK，但也可能扩展到仓库的子目录，例如 MCP 文档。维护者可以自行制定决策规则和程序。维护者应独立地对其负责的项目做出决策，但在必要时可以将问题推迟或升级到核心维护者。

维护者的职责包括：

- 与社区贡献者进行有思考性和富有成效的互动，
- 维护并改进其负责的 MCP 项目部分，
- 支持文档、路线图及其他相关的 MCP 项目部分，
- 将社区的想法传达给核心团队。

在需要时，维护者可提名额外的维护者。维护者只能由核心维护者或首席核心维护者随时任命或移除，无需说明理由。

维护者对其负责的仓库拥有写入和/或管理员权限。

### 核心维护者

核心维护者应深入理解 Model Context Protocol 及其规范。其职责包括：

- 设计、审查和引导 MCP 规范的演进，以及 MCP 项目的其他部分（如文档），
- 阐述项目的一致长期愿景，
- 以公平和透明的方式调解和解决争议问题，在可能的情况下寻求共识，必要时做出果断决策，
- 任命或移除维护者，
- 以 MCP 的最佳利益为出发点管理 MCP 项目。

核心维护者作为一个整体，有权通过多数投票否决维护者做出的任何决定。核心维护者有权根据自己的判断解决争议。核心维护者应公开阐明他们的决策过程。核心小组负责制定自己的决策程序。

核心维护者通常拥有对所有 MCP 仓库的写入和管理员权限，但应使用与外部贡献者相同的提交机制（通常是拉取请求）。出于安全考虑时可作出例外处理。

### 首席维护者（BDFL）

MCP 有两位首席维护者：Justin Spahr-Summers 和 David Soria Parra。首席维护者可以否决核心维护者或维护者的任何决定。这种模型在开源社区中通常被称为“仁慈的终身独裁者”（Benevolent Dictator for Life，简称 BDFL）。首席维护者应公开阐明他们的决策，并给出明确的决策理由。首席维护者属于核心维护者组的一部分。

首席维护者负责确认或移除核心维护者。

首席维护者尽可能地是 MCP 项目所有基础设施的管理员。这包括但不限于所有通信渠道、GitHub 组织和仓库。

### 决策流程

核心维护者小组每两周举行一次会议，讨论并投票表决提案，以及讨论任何需要的议题。如有需要，可通过共享的 Discord 服务器讨论和投票较小的提案。

首席维护者、核心维护者和维护者小组应每三到六个月尽量举行一次线下会议。

## 流程

核心维护者和首席维护者负责 Model Context Protocol 的所有方面，包括文档、问题、内容建议以及 [MCP 项目](https://github.com/modelcontextprotocol) 下的所有其他部分。维护者负责其负责区域的文档、问题和内容建议，但鼓励他们参与 MCP 项目的整体维护。维护者、核心维护者和首席维护者应使用与外部贡献者相同的贡献流程，而非直接修改仓库。这有助于阐明意图并提供讨论机会。

### 项目和工作组

MCP 项目分为两个主要结构：项目和工作组。

项目是维护在专用仓库中的具体组件。这些包括规范、TypeScript SDK、Go SDK、检查器以及其他实现工件。

工作组是感兴趣方协作的论坛，用于讨论 MCP 的特定方面，而不维护代码仓库。这些包括专注于传输协议、客户端实现以及其他跨领域问题的小组。

#### 治理原则

所有项目和工作组在遵守以下核心原则的前提下实行自治：

1. 明确的贡献和决策流程
2. 开放的沟通和透明的决策

两者都必须：

- 记录其贡献流程
- 保持透明的沟通
- 公开做出决策（工作组必须发布会议纪要和提案）

没有指定流程的项目和工作组默认使用：

- GitHub 拉取请求和问题用于贡献
- 官方 [MCP 贡献者 Discord](/community/communication#discord) 中的公开频道

#### 维护职责

没有专门维护者的组件（如文档）由核心维护者负责。这些组件遵循通过拉取请求的标准贡献指南，由维护者处理审查，并在有重大更改时升级到核心维护者审查。

鼓励核心维护者和维护者改进 MCP 项目的任何部分，无论其正式维护职责如何。

### 规范项目

#### 规范增强提案（SEP）

对规范的修改提议必须以书面形式提出，从提案摘要开始，阐述其要解决的 **问题**，提出 **解决方案**、**替代方案**、**考虑因素、结果** 和 **风险**。[SEP 指南](/community/sep-guidelines) 提供了关于 SEP 应有结构的信息。SEP 应在 [规范仓库](https://github.com/modelcontextprotocol/specification) 中作为问题创建，并打上 `proposal, sep` 标签。

所有提案必须由 MCP 指导小组（维护者、核心维护者或首席核心维护者）中的一名 **发起人** 提出。发起人负责确保提案得到积极开发，符合提案的质量标准，并负责在核心维护者会议上提出和讨论。维护者和核心维护者小组应定期审查未有发起人的开放提案。如果提案在六个月内未能找到发起人，则自动被拒绝。

一旦提案有了发起人，它将被分配给该发起人，并打上 `draft` 标签。

## 沟通

### 核心维护者会议

核心维护者小组每两周举行一次会议，讨论提案和项目。提案的会议纪要应公开。核心维护者小组应每 3-6 个月尽量举行一次线下会议。

### 公共聊天

MCP 项目维护着一个 [公共 Discord 服务器](/community/communication#discord)，设有面向兴趣小组的开放聊天。MCP 项目也可能为某些通信设置私有频道。

## 提名、确认和移除维护者

### 原则

- 模块维护者组的成员资格基于个人贡献，授予那些通过贡献、评审和讨论展现出其领域专业知识并对 MCP 的整体方向有高度认同的个人。
- 要成为 **维护者** 组成员，个人必须持续展现出对 MCP 总体原则的高度认同。
- 模块维护者或核心维护者无任期限制。
- 如果工作组或子项目维护者长期不活跃，可将其状态调整为“名誉”状态，具体标准由各维护者组自行定义。
- 成员资格属于个人，而非公司。

### 提名和移除

- 核心维护者负责添加和移除维护者。他们将参考现有维护者的意见。
- 首席维护者负责添加和移除核心维护者。
